Hybrid Flow和Offline Access

目前我們解決方案里面有三項目Authorization Server,Web api和Mvc Client,在現實世界中,他們都在不同的地方
現在讓我們從MvcClient使用Authorization Server獲取的token來訪問web api.併且確保這個token不過期
現在我們的MvcClient使用的是implicit flow,也就是說，token被發送到client,這種情況下token的生命可能短，但
是我們可以重向Authorization server重新獲取新的token.
例如，SPA(Single Page Application)中,implicit flow基本上就是除了Resource owner password flow以外唯一合
適的flow,但是我們的網站可能會在client(SPA client/或者指用戶)沒有使用網站訪問api,為了這樣做，不但要保證
token不過期，我們還需要使用別的flow,我們要介紹一下authorization code flow,它和implicit flow很像，不同的
是在重定向回到網站的時候獲取的不是access token,而是從authorization server獲取一個code,使用它網站可以交換
一個sercret,使用這個sercret可以獲取access token和reftesh token

Hybrid flow,是兩䅜的混合，首先identity token通過瀏覽器傳過來了，然後客戶端可以在進行任何工作之前對其驗證
，如果驗證成功，客戶端就會再打開一個通道向Authorization Server請求獲取access token

GrantType要改成Hybrid或是HybrdAndClientCredentials，如果只使用Code Flow的話不行，因為我們的網站使用
Authorization Server來進行Authentication，我們想獲取Access token以便被授權來訪問api,所以這里用HybridFlow
還要添加一個新的Email scope, 因為我想改變api來允許我基于email來創建用戶的數據，因為authorization server和
web api是分開的，所以用戶的數據庫也是分開的，api使用用戶名(email)來查詢數據蟀中的數據
AllowOfflineAccess我們還需要獲取Refresh Token，這就要求我們的網站必須可以”離線”工作，這里離線是指用戶和網站
之間斷開了，併不是指網站離線了
這就是說網站可以使用token來和api進行交互，而不需要用戶登入到網站上

好，太麻煩了，既然我們有refresh token了，那不如向authorization server請求一個新的access token和refresh token,
然後再把這些更新到cookie里面， 所以下次再調api的時候使用的是新的token
